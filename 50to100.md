## 51. What is the Virtual DOM, and how does React's Reconciliation (Diffing) algorithm work?

**Answer:**
The Virtual DOM is a lightweight JavaScript representation of the actual DOM. React uses it to improve performance by minimizing direct DOM manipulations.

**Reconciliation Algorithm:**
1. **Render Phase:** React creates a Virtual DOM tree from components.
2. **Diffing:** When state/props change, React creates a new Virtual DOM tree and compares it with the previous one using a diffing algorithm.
3. **Key Assumptions:**
   - Elements of different types produce different trees.
   - Stable keys help identify which items have changed, been added, or removed.
4. **Reconciliation Process:**
   - If root elements differ, tear down and rebuild.
   - If same type, compare attributes and update only changed ones.
   - For children, iterate over both lists and apply differences (keyed).
5. **Commit Phase:** Apply the minimal set of changes to the real DOM.

**Example:**
```jsx
// Virtual DOM representation
const vdom = {
  type: 'div',
  props: { className: 'container' },
  children: [
    { type: 'h1', props: {}, children: ['Hello'] }
  ]
};

// Diffing: when state changes, React compares new and old VDOM trees
```

52. What is the difference between Element and Component in React?

Answer:

· Element: A plain object describing what you want to see on the screen. Created using React.createElement() or JSX. Immutable.
· Component: A function or class that accepts props and returns a React element. Can be stateful and have lifecycle methods.

Example:

```jsx
// Element (JSX)
const element = <h1>Hello, world!</h1>;

// Component (functional)
function Welcome(props) {
  return <h1>Hello, {props.name}</h1>;
}

// Component (class-based)
class Welcome extends React.Component {
  render() {
    return <h1>Hello, {this.props.name}</h1>;
  }
}
```

53. Explain the strict architectural rules regarding Props vs State.

Answer:

Aspect Props State
Mutability Immutable (read-only) Mutable (can be updated)
Ownership Owned by parent component Owned by component itself
Source Passed from parent Defined/initialized inside component
Update Only changed by parent Changed via setState/useState
Purpose Configuring child components Managing dynamic data within component

Rules:

· Never modify props directly; they are read-only.
· State should be private to the component.
· State updates may be asynchronous; use functional updates when relying on previous state.
· Lift state up when multiple components need to share data.

```jsx
// Props (read-only)
function Child({ name }) {
  // name cannot be changed here
  return <div>{name}</div>;
}

// State (mutable)
function Counter() {
  const [count, setCount] = useState(0);
  return <button onClick={() => setCount(c => c + 1)}>{count}</button>;
}
```

54. Why should you never mutate state directly (e.g., this.state.x = y or state = y)?

Answer:
Direct mutation bypasses React's reactivity system, preventing:

· Component re-rendering (won't trigger updates)
· Correct lifecycle method calls
· Efficient batching of updates
· Immutability guarantees for PureComponent/React.memo optimizations

Correct approach: Use setState (class) or setter from useState (functional).

```jsx
// ❌ Wrong
this.state.count = 5;
this.setState({ count: this.state.count + 1 }); // may still work but unsafe

// ✅ Correct
this.setState({ count: 5 });
// or functional update
this.setState(prevState => ({ count: prevState.count + 1 }));

// Functional component
const [count, setCount] = useState(0);
setCount(5);
setCount(prev => prev + 1);
```

55. What is "Props Drilling", and what are three distinct architectural ways to solve it?

Answer:
Props Drilling: Passing data down multiple component levels through props even when intermediate components do not use the data.

Solutions:

1. Context API: Provides a way to share values without passing props manually.
2. State Management Libraries: Redux, MobX, Zustand.
3. Component Composition: Use render props or children props to pass data only to components that need it.

Example:

```jsx
// ❌ Props Drilling
<GrandParent user={user}>
  <Parent user={user}>
    <Child user={user} />
  </Parent>
</GrandParent>

// ✅ Using Context
const UserContext = React.createContext();
<GrandParent>
  <UserContext.Provider value={user}>
    <Parent>
      <Child />
    </Parent>
  </UserContext.Provider>
</GrandParent>

// In Child: const user = useContext(UserContext);

// ✅ Using Redux (connect/useSelector)
const Child = () => {
  const user = useSelector(state => state.user);
  // ...
}

// ✅ Component Composition
<GrandParent>
  <Parent renderChild={() => <Child user={user} />} />
</GrandParent>
```

57. What is the purpose of the key prop in React lists? What happens if you use Math.random() or array index as a key?

Answer:
The key prop helps React identify which items have changed, been added, or removed, enabling efficient reconciliation.

· Stable key: Should be unique, persistent, and predictable (e.g., item ID).
· Using array index: Causes issues when reordering, adding, or removing items; leads to bugs with component state and poor performance.
· Using Math.random(): Changes on every render, causing React to re-render all items unnecessarily, losing state and performance.

Example:

```jsx
// ❌ Bad: Using index as key
{items.map((item, index) => <Item key={index} {...item} />)}

// ❌ Bad: Using Math.random()
{items.map(item => <Item key={Math.random()} {...item} />)}

// ✅ Good: Using a unique identifier
{items.map(item => <Item key={item.id} {...item} />)}
```

58. What are Higher-Order Components (HOC), and how do they differ from Render Props?

Answer:

· HOC: A function that takes a component and returns a new component with enhanced behavior. Common for cross-cutting concerns (authentication, logging).
· Render Props: A technique where a component receives a function prop that returns a React element, giving the parent control over rendering.

HOC Example:

```jsx
function withAuth(WrappedComponent) {
  return function WithAuth(props) {
    const isAuth = useAuth();
    if (!isAuth) return <Login />;
    return <WrappedComponent {...props} />;
  };
}
const ProtectedProfile = withAuth(Profile);
```

Render Props Example:

```jsx
<DataFetcher url="/api/users">
  {({ data, loading }) => (loading ? <Spinner /> : <UserList users={data} />)}
</DataFetcher>
```

Differences:

· HOC composes components, render props invert control.
· Render props are more flexible for sharing behavior without wrapper components.
· HOCs can cause prop name collisions; render props avoid that.

59. How does React handle events? What is SyntheticEvent?

Answer:
React uses a SyntheticEvent wrapper around native browser events to ensure cross-browser compatibility.

Key points:

· Events are normalized across browsers.
· Events are pooled (reused) for performance; event properties can be accessed asynchronously.
· Most events are delegated to the root (document) – event delegation.
· SyntheticEvent provides the same interface as native events but with consistent properties.

Example:

```jsx
function handleClick(e) {
  e.preventDefault(); // works across browsers
  console.log(e.target);
  // To access event asynchronously:
  // e.persist(); // in older React, not needed in modern versions
}
<button onClick={handleClick}>Click</button>
```

61. Explain the concept of "Lifting State Up" with a practical layout example.

Answer:
Lifting state up means moving shared state to the closest common ancestor so that multiple components can access and update it.

Example: Temperature Converter

```jsx
// Child components (stateless)
function FahrenheitInput({ temperature, onTemperatureChange }) {
  return <input value={temperature} onChange={e => onTemperatureChange(e.target.value)} />;
}

function CelsiusInput({ temperature, onTemperatureChange }) {
  return <input value={temperature} onChange={e => onTemperatureChange(e.target.value)} />;
}

// Parent component (holds state)
function TemperatureCalculator() {
  const [temperature, setTemperature] = useState('');
  const [scale, setScale] = useState('c'); // 'c' or 'f'

  const handleCelsiusChange = (value) => {
    setTemperature(value);
    setScale('c');
  };
  const handleFahrenheitChange = (value) => {
    setTemperature(value);
    setScale('f');
  };

  const celsius = scale === 'f' ? tryConvert(temperature, toCelsius) : temperature;
  const fahrenheit = scale === 'c' ? tryConvert(temperature, toFahrenheit) : temperature;

  return (
    <div>
      <CelsiusInput temperature={celsius} onTemperatureChange={handleCelsiusChange} />
      <FahrenheitInput temperature={fahrenheit} onTemperatureChange={handleFahrenheitChange} />
    </div>
  );
}
```

62. What is the purpose of React Error Boundaries? Which lifecycle methods are required to implement them?

Answer:
Error Boundaries are React components that catch JavaScript errors anywhere in their child component tree, log them, and display a fallback UI instead of crashing the whole app.

Methods required (class components):

· static getDerivedStateFromError(error) – updates state to show fallback UI.
· componentDidCatch(error, errorInfo) – logs error information.

Usage:

```jsx
class ErrorBoundary extends React.Component {
  constructor(props) {
    super(props);
    this.state = { hasError: false };
  }

  static getDerivedStateFromError(error) {
    return { hasError: true };
  }

  componentDidCatch(error, errorInfo) {
    logErrorToService(error, errorInfo);
  }

  render() {
    if (this.state.hasError) {
      return <h1>Something went wrong.</h1>;
    }
    return this.props.children;
  }
}

// Use
<ErrorBoundary>
  <MyComponent />
</ErrorBoundary>
```

Note: Error Boundaries only catch errors in the component tree below them, not in event handlers or asynchronous code.

Answer:
React.StrictMode is a tool that highlights potential problems in an application. It runs additional checks and warnings in development mode.

Why double rendering:

· To detect side effects that break the rule of pure components.
· StrictMode intentionally invokes certain lifecycle methods and render functions twice to help detect accidental side effects.
· This only happens in development; production builds are unaffected.

Benefits:

· Identifies unsafe lifecycles.
· Warns about legacy string refs.
· Detects unexpected side effects.
· Detects deprecated findDOMNode usage.

Usage:

```jsx

```

64. State the core "Rules of Hooks" that every React developer must follow.

Answer:

1. Only Call Hooks at the Top Level: Not inside loops, conditions, or nested functions. This ensures hooks are called in the same order on every render, preserving state.
2. Only Call Hooks from React Functions: Either from React function components or custom hooks. Not from regular JavaScript functions.
3. Use Hooks at the Top Level of Your Custom Hooks: Custom hooks should also call other hooks unconditionally.

Reason: React relies on the order of hooks to associate state correctly.

65. How do you replicate componentDidMount, componentDidUpdate, and componentWillUnmount using useEffect?

Answer:

```jsx
useEffect(() => {
  // componentDidMount
  console.log('Component mounted');
  
  return () => {
    // componentWillUnmount
    console.log('Component unmounted');
  };
}, []); // empty deps => run once on mount and cleanup on unmount

// componentDidUpdate (runs on every render)
useEffect(() => {
  console.log('Component updated');
}); // no deps => runs after every render

// componentDidUpdate with specific dependencies
useEffect(() => {
  console.log('Count changed');
}, [count]); // runs when 'count' changes

// For mounting + updates

  // Runs on mount and on every update
  console.log('Mounted or updated');
});
```

66. What happens if you omit the dependency array in a useEffect hook? What if it is empty []?

Answer:

· Omit deps: Effect runs after every render (both mount and update).
· Empty array []: Effect runs only once after initial mount (componentDidMount) and cleanup on unmount.
· With dependencies: Effect runs whenever any dependency changes (after mount and subsequent updates).

67. How do you clean up side effects (like intervals or event listeners) inside a useEffect hook?

Answer:
Return a cleanup function from the effect. React will run it before the component unmounts or before re-running the effect.

Example:

```jsx
useEffect(() => {
  const timer = setInterval(() => {
    console.log('Tick');
  }, 1000);

  const handleResize = () => console.log(window.innerWidth);
  window.addEventListener('resize', handleResize);

  return () => {
    clearInterval(timer);
    window.removeEventListener('resize', handleResize);
  };
}, []);
```

68. What is the difference between useMemo and useCallback? Provide a concrete use case for both.

Answer:

· useMemo memoizes the result of a computation. It returns a memoized value.
· useCallback memoizes a function instance. Returns a memoized callback.

Use Cases:

· useMemo: Expensive calculations, e.g., filtering a large list.
· useCallback: Passing stable callbacks to child components to prevent unnecessary re-renders (especially with React.memo).

Example:

```jsx
// useMemo
const expensiveResult = useMemo(() => {
  return computeExpensiveValue(a, b);
}, [a, b]);

// useCallback
const handleClick = useCallback(() => {
  doSomething(a, b);
}, [a, b]);

// Child component uses React.memo
const Child = React.memo(({ onClick }) => {
  // will re-render only if onClick changes
  return <button onClick={onClick}>Click</button>;
});
```

69. How does the useRef hook work? Name two scenarios where you would choose it over useState.

Answer:
useRef returns a mutable ref object whose .current property persists across renders and does not cause re-renders when mutated.

Scenarios to use useRef over useState:

1. Storing DOM references (e.g., focusing an input).
2. Keeping a mutable value that doesn't trigger re-renders (e.g., timer IDs, previous value tracking).

Example:

```jsx
function MyComponent() {
  const inputRef = useRef(null);
  const timerRef = useRef(null);

  const focusInput = () => {
    inputRef.current.focus();
  };

  useEffect(() => {
    timerRef.current = setInterval(() => console.log('tick'), 1000);
    return () => clearInterval(timerRef.current);
  }, []);

  return <input ref={inputRef} />;
}
```

70. What is the purpose of the useReducer hook? When should you use it instead of multiple useState hooks?

Answer:
useReducer is a hook that manages complex state logic that involves multiple sub-values or when the next state depends on the previous state in a non-trivial way.

When to use:

· State has complex transitions (e.g., form with many fields).
· Next state depends on previous state.
· State management logic is best encapsulated outside the component.

Example:

```jsx
const initialState = { count: 0, step: 1 };

function reducer(state, action) {
  switch (action.type) {
    case 'increment':
      return { ...state, count: state.count + state.step };
    case 'decrement':
      return { ...state, count: state.count - state.step };
    case 'setStep':
      return { ...state, step: action.payload };
    default:
      return state;
  }
}

function Counter() {
  const [state, dispatch] = useReducer(reducer, initialState);
  return (
    <div>
      Count: {state.count}
      <button onClick={() => dispatch({ type: 'increment' })}>+</button>
      <button onClick={() => dispatch({ type: 'decrement' })}>-</button>
      <input value={state.step} onChange={e => dispatch({ type: 'setStep', payload: Number(e.target.value) })} />
    </div>
  );
}
```

71. How do you create a Custom Hook? Write a conceptual outline for a useFetch hook.

Answer:
A custom hook is a JavaScript function whose name starts with "use" and that may call other hooks.

Example: useFetch

```jsx
import { useState, useEffect } from 'react';

function useFetch(url, options = {}) {
  const [data, setData] = useState(null);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState(null);

  useEffect(() => {
    const abortController = new AbortController();
    setLoading(true);
    setError(null);

    fetch(url, { ...options, signal: abortController.signal })
      .then(res => {
        if (!res.ok) throw new Error('Network response was not ok');
        return res.json();
      })
      .then(data => {
        setData(data);
        setLoading(false);
      })
      .catch(err => {
        if (err.name !== 'AbortError') {
          setError(err.message);
          setLoading(false);
        }
      });

    return () => abortController.abort();
  }, [url]);

  return { data, loading, error };
}
```

73. Explain the useLayoutEffect hook. How does it differ from the standard useEffect hook?

Answer:

· useEffect runs asynchronously after the browser has painted (layout and paint phases). It does not block the browser's painting.
· useLayoutEffect runs synchronously after all DOM mutations but before the browser paints. It blocks painting, so it's useful for reading DOM layout and performing synchronous updates that need to be applied before the user sees the changes.

When to use useLayoutEffect:

· Measuring DOM nodes (e.g., dimensions).
· Synchronously adjusting DOM to avoid flicker.
· Operations that must happen before the browser paints.

Caution: Overuse can cause performance issues because it blocks painting.

75. How does React.memo() optimize component rendering performance?

Answer:
React.memo is a higher-order component that memoizes the rendered result based on props. It performs a shallow comparison of props; if props haven't changed, the component reuses the last rendered result, skipping re-renders.

Usage:

```jsx
const MemoizedComponent = React.memo(function MyComponent(props) {
  // render
});

// Custom comparison
React.memo(Component, (prevProps, nextProps) => {
  return prevProps.id === nextProps.id; // return true to skip update
});
```

Note: Only works for functional components; class components use PureComponent or shouldComponentUpdate.

76. What are the primary causes of memory leaks in React applications, and how do you fix them?

Answer:

1. Uncleaned subscriptions/event listeners: Use cleanup functions in useEffect.
2. Timers/intervals not cleared: clearInterval/clearTimeout in cleanup.
3. Unmounted component trying to update state: Use a flag (e.g., isMounted) or abort controller.
4. Large data structures held in state that are not garbage-collected: Manage state carefully and unmount properly.
5. Closures holding references to large objects: Avoid capturing large objects in closures unnecessarily.

Fix examples:

```jsx
useEffect(() => {
  const timer = setInterval(() => { /* ... */ }, 1000);
  return () => clearInterval(timer);
}, []);

useEffect(() => {
  let isMounted = true;
  fetchData().then(data => {
    if (isMounted) setData(data);
  });
  return () => { isMounted = false; };
}, []);
```

80. How does state batching work in React 18?

Answer:
React 18 introduced automatic batching of state updates across events, promises, setTimeout, and native event handlers. This reduces unnecessary re-renders by grouping multiple state updates into a single render.

Before React 18: Only updates inside React event handlers were batched.
After React 18: Updates inside promises, setTimeout, and other asynchronous callbacks are also batched.

Example:

```jsx
function handleClick() {
  setCount(c => c + 1);
  setFlag(f => !f);
  // React batches these two updates, re-renders once.
}
```

Opt-out: Use flushSync from react-dom to force synchronous updates.

82. Explain the core data flow architecture of Redux (Action -> Dispatcher -> Store -> Reducer -> View).

Answer:

1. Action: A plain object describing what happened ({ type: 'INCREMENT' }).
2. Dispatch: Sends the action to the store.
3. Store: Holds the state tree and dispatches actions.
4. Reducer: A pure function that takes previous state and action, returns new state.
5. View: React components subscribe to store and re-render on changes.

Diagram:

```
View —> Action —> Dispatch —> Reducer —> Store —> View (update)
```

Example:

```jsx
// Action
const increment = () => ({ type: 'INCREMENT' });

// Reducer
function counter(state = { count: 0 }, action) {
  switch (action.type) {
    case 'INCREMENT': return { count: state.count + 1 };
    default: return state;
  }
}

// Store (using Redux)
const store = createStore(counter);
store.dispatch(increment());
```

83. What is a "pure function," and why must Redux reducers strictly be pure functions?

Answer:
A pure function:

· Given the same inputs, always returns the same output.
· Has no side effects (does not mutate arguments, no API calls, no random values).

Redux reducers must be pure because:

· They enable predictable state updates.
· They allow time-travel debugging.
· They make testing easy.
· They prevent bugs due to side effects.

Example of impure reducer (❌):

```jsx
function reducer(state, action) {
  state.value += 1; // mutation
  return state;
}
```

Pure reducer (✅):

```jsx
function reducer(state, action) {
  return { ...state, value: state.value + 1 };
}
```

85. What is Redux Thunk, and how does it handle asynchronous API actions?

Answer:
Redux Thunk is a middleware that allows action creators to return a function instead of an action object. That function receives dispatch and getState, enabling asynchronous logic.

Example:

```jsx
// Action creator with thunk
const fetchUser = (userId) => async (dispatch, getState) => {
  dispatch({ type: 'FETCH_USER_REQUEST' });
  try {
    const response = await api.getUser(userId);
    dispatch({ type: 'FETCH_USER_SUCCESS', payload: response.data });
  } catch (error) {
    dispatch({ type: 'FETCH_USER_FAILURE', error: error.message });
  }
};

// Usage: dispatch(fetchUser(123))
```

86. How do useSelector and useDispatch hooks connect a React component to the Redux store?

```jsx
import { useSelector, useDispatch } from 'react-redux';

  );
```

87. What is the purpose of Redux middleware? Name a common use case for it.

Answer:
Middleware provides a third-party extension point between dispatching an action and the moment it reaches the reducer. It enables:

· Logging, crash reporting, asynchronous API calls, routing, etc.

Common use cases:

· Redux Thunk/Saga: Handling async actions.
· Redux Logger: Logging actions and state.
· Redux Persist: Persisting state to localStorage.

Example middleware:

```jsx
const logger = store => next => action => {
  console.log('dispatching', action);
  let result = next(action);
  console.log('next state', store.getState());
  return result;
};
```

97. What is the difference between fireEvent and userEvent in React Testing Library?

Answer:

· fireEvent: Dispatches DOM events immediately. Simulates single events, but may not mimic full browser interactions (like focus/blur for input).
· userEvent: Simulates full user interactions (clicks, typing, hover) more realistically, including focus, keypress, etc.

Example:

```jsx
// fireEvent - simple click
fireEvent.click(button);

// userEvent - more realistic
await userEvent.click(button);
```

userEvent is recommended for user interaction testing.

99. What is Server-Side Rendering (SSR), and how does it fundamentally differ from Client-Side Rendering (CSR)?

Answer:

· Client-Side Rendering (CSR): HTML is minimal; JavaScript renders the UI in the browser. Initial load is slower; SEO is poorer; but subsequent interactions are fast.
· Server-Side Rendering (SSR): HTML is rendered on the server and sent to the client. Initial load is faster; SEO is better; but server load is higher, and hydration is required.

CSR flow: Browser loads bare HTML -> downloads JS -> executes -> renders content.
SSR flow: Server renders HTML with data -> sends to browser -> browser displays fully rendered page -> hydrates with React.

100. What is the purpose of .env files in a React project, and how do you expose variables securely?

Answer:
.env files store environment-specific configuration (API URLs, keys). They are used to avoid hardcoding values and to keep secrets out of version control.

Exposing variables:

· In Create React App and Vite, variables prefixed with REACT_APP_ (CRA) or VITE_ (Vite) are embedded into the build.
· They are not truly secure; they are accessible in the browser source. Never store sensitive secrets like API keys in frontend .env. Use server-side proxies for sensitive credentials.

Example:

```env
# .env
REACT_APP_API_URL=https://api.example.com
VITE_API_URL=https://api.example.com
```

Usage:

```jsx
const apiUrl = process.env.REACT_APP_API_URL;
// or Vite: import.meta.env.VITE_API_URL
```
