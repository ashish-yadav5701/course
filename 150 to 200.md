# Topic 4: Database & SQL Queries (Questions 151–200)

## 151. Explain the differences between the 1st, 2nd, and 3rd Normal Forms (1NF, 2NF, 3NF) in database normalization.

**Answer:**
Normalization organizes data to reduce redundancy and improve integrity.

- **1NF (First Normal Form):**  
  - Each table cell contains a single atomic value (no repeating groups or arrays).  
  - Each column contains values of the same type.  
  - Each record is uniquely identified by a primary key.

- **2NF (Second Normal Form):**  
  - Must be in 1NF.  
  - No partial dependency – every non‑prime attribute must depend on the entire primary key (relevant for composite keys).  
  - All columns must depend on the whole key, not part of it.

- **3NF (Third Normal Form):**  
  - Must be in 2NF.  
  - No transitive dependency – non‑prime attributes must not depend on other non‑prime attributes; they must depend only on the primary key.

**Example (2NF violation):**  
`Order (OrderID, ProductID, ProductName, Quantity)` – `ProductName` depends only on `ProductID`, not on `OrderID` (partial dependency). Split into `Orders` and `Products`.

**3NF violation:**  
`Employee (EmpID, DeptID, DeptName)` – `DeptName` depends on `DeptID`, which is not the primary key. Move `DeptName` to a separate `Department` table.

## 152. What is Denormalization? Under what production scenarios would you intentionally denormalize a schema?

**Answer:**
Denormalization is the process of intentionally adding redundancy to a normalized schema to improve read performance. It merges tables, adds duplicate columns, or pre‑computes aggregates.

**Scenarios for denormalization:**
- Read-heavy applications (e.g., dashboards, reporting) where joins are expensive.
- When you need to reduce query complexity and latency.
- In caching layers (e.g., storing pre‑joined data in Redis).
- When using NoSQL databases (where denormalization is the norm).

**Trade‑offs:** Increased storage, complex updates (need to keep duplicates in sync), and higher write overhead.

## 153. Explain the ACID properties of a database transaction with real-world examples.

**Answer:**
ACID ensures reliable processing of transactions:

- **Atomicity:** All operations succeed or none do.  
  *Example:* Money transfer – debit from account A and credit to account B; if either fails, the whole transaction rolls back.

- **Consistency:** Transaction brings database from one valid state to another, preserving all rules (constraints, triggers).  
  *Example:* A transaction that reduces inventory must not result in negative stock if a check constraint exists.

- **Isolation:** Concurrent transactions execute as if they were sequential.  
  *Example:* Two users booking the last seat – isolation ensures one transaction sees the seat as available and the other waits until the first commits.

- **Durability:** Once committed, changes persist even after a crash.  
  *Example:* After transferring money, the updated balances survive a power outage.

## 154. What are the four primary Transaction Isolation Levels? Detail the specific problems they solve.

**Answer:**

| Isolation Level | Dirty Read | Non‑repeatable Read | Phantom Read |
|-----------------|------------|---------------------|--------------|
| Read Uncommitted | ❌ Allowed  | ❌ Possible          | ❌ Possible   |
| Read Committed  | ✅ Prevent  | ❌ Possible          | ❌ Possible   |
| Repeatable Read | ✅ Prevent  | ✅ Prevent           | ❌ Possible   |
| Serializable    | ✅ Prevent  | ✅ Prevent           | ✅ Prevent    |

- **Dirty Read:** Reading uncommitted data from another transaction (e.g., rolling back later).
- **Non‑repeatable Read:** A row’s value changes between two reads within the same transaction.
- **Phantom Read:** New rows appear (or disappear) between two queries due to insert/delete by another transaction.

## 155. What is the difference between a Primary Key, a Unique Key, and a Foreign Key?

**Answer:**
- **Primary Key:** Uniquely identifies each row; must be NOT NULL and unique. Only one per table.
- **Unique Key:** Ensures all values in a column (or combination) are distinct; allows a single NULL (in SQL Server) or multiple NULLs (depending on DB). Many per table.
- **Foreign Key:** Enforces referential integrity by linking a column (or columns) to a primary/unique key in another table. Allows NULLs; prevents orphan records.

## 156. Explain Referential Integrity constraints. What is the difference between ON DELETE CASCADE and ON DELETE SET NULL?

**Answer:**
Referential integrity ensures that a foreign key value always points to an existing row in the referenced table.

- **ON DELETE CASCADE:** When the parent row is deleted, all child rows referencing it are automatically deleted.
- **ON DELETE SET NULL:** When the parent row is deleted, the foreign key column(s) in child rows are set to NULL (if allowed). This preserves child rows but breaks the reference.

## 157. What are the engineering tradeoffs between using a Relational Database (SQL) vs a Non-Relational Database (NoSQL)?

**Answer:**

| Aspect | SQL (Relational) | NoSQL (Non‑relational) |
|--------|------------------|------------------------|
| **Schema** | Fixed, rigid schema; migrations required | Flexible, dynamic schema; easy to evolve |
| **Scalability** | Vertical scaling (hard); horizontal via sharding (complex) | Horizontal scaling (built‑in) |
| **ACID** | Strong ACID support | Often eventual consistency (BASE) |
| **Joins** | Powerful joins | Limited or no joins (denormalized) |
| **Use Cases** | Complex queries, transactions, reporting | High‑volume data, caching, real‑time apps |
| **Maturity** | Mature, standardized | Diverse, evolving |

**Trade‑off:** Choose SQL for data integrity and complex queries; choose NoSQL for scalability, flexible schema, and high throughput.

## 158. Write a generic SQL query to find the Nth highest salary from an Employee table without using vendor‑specific keywords (like LIMIT or TOP).

**Answer:**
Generic approach using a correlated subquery:

```sql
SELECT DISTINCT e1.salary
FROM Employee e1
WHERE N-1 = (
    SELECT COUNT(DISTINCT e2.salary)
    FROM Employee e2
    WHERE e2.salary > e1.salary
);
```

Alternatively, using ROW_NUMBER() (standard SQL):

```sql
WITH Ranked AS (
    SELECT salary, DENSE_RANK() OVER (ORDER BY salary DESC) AS rnk
    FROM Employee
)
SELECT DISTINCT salary
FROM Ranked
WHERE rnk = N;
```

159. What is the difference between WHERE clauses and HAVING clauses? Provide an example query using both.

Answer:

· WHERE: Filters rows before grouping; cannot contain aggregate functions.
· HAVING: Filters groups after grouping; can contain aggregate functions.

Example:

```sql
SELECT department, AVG(salary) AS avg_salary
FROM Employee
WHERE hire_date >= '2020-01-01'
GROUP BY department
HAVING AVG(salary) > 60000;
```

This filters employees hired after 2020, groups by department, and keeps only those departments with average salary above 60,000.

160. Explain the differences between INNER JOIN, LEFT JOIN, RIGHT JOIN, and FULL OUTER JOIN.

Answer:

· INNER JOIN: Returns rows where there is a match in both tables.
· LEFT JOIN (or LEFT OUTER): Returns all rows from the left table, with matching rows from the right; non‑matching right side are NULL.
· RIGHT JOIN (or RIGHT OUTER): Returns all rows from the right table, with matching rows from the left; non‑matching left side are NULL.
· FULL OUTER JOIN: Returns all rows from both tables, with NULLs where no match exists.

162. Write a query to find and delete all duplicate rows in a table while keeping only the row with the lowest unique ID.

Answer:
Assuming a table Users with columns id, email, and duplicates defined by email:

```sql
DELETE FROM Users
WHERE id NOT IN (
    SELECT MIN(id)
    FROM Users
    GROUP BY email
);
```

For large tables, use a CTE with ROW_NUMBER():

```sql
WITH Duplicates AS (
    SELECT id, ROW_NUMBER() OVER (PARTITION BY email ORDER BY id) AS rn
    FROM Users
)
DELETE FROM Users
WHERE id IN (SELECT id FROM Duplicates WHERE rn > 1);
```

163. Explain the functionality of SQL Window Functions. Write a query using ROW_NUMBER(), RANK(), and DENSE_RANK(), and explain the differences.

Answer:
Window functions perform calculations across a set of rows related to the current row, without collapsing them.

· ROW_NUMBER(): Assigns a unique sequential number to each row within a partition, starting at 1. Ties get different numbers.
· RANK(): Assigns the same number to tied rows, but skips numbers after ties (e.g., 1,2,2,4).
· DENSE_RANK(): Similar to RANK but does not skip numbers (e.g., 1,2,2,3).

Example:

```sql
SELECT employee_id, salary,
       ROW_NUMBER() OVER (ORDER BY salary DESC) AS row_num,
       RANK()       OVER (ORDER BY salary DESC) AS rank,
       DENSE_RANK() OVER (ORDER BY salary DESC) AS dense_rank
FROM Employee;
```

164. What is a Common Table Expression (CTE)? How does it compare to a Subquery or a Temporary Table regarding readability and scope?

Answer:
A CTE (WITH clause) defines a temporary named result set that exists only within the scope of a single query.

Comparison:

· CTE vs Subquery: CTEs improve readability, especially with multiple references or recursive queries. Subqueries are nested and can become messy.
· CTE vs Temporary Table: A temp table is a physical table that persists for the session; it can be indexed and reused across multiple queries. CTEs are in‑memory and exist only for the duration of the query, cannot be indexed, and are generally lighter.

168. How do you use the EXISTS operator versus the IN operator? Which performs better with a large subquery dataset?

Answer:

· IN: Checks if a value matches any value in a subquery result; the subquery is fully evaluated.
· EXISTS: Tests for the existence of rows in a subquery; it stops processing once a match is found.

Performance: For large subquery result sets, EXISTS is often faster because it can use an index and stops early. IN may be optimized similarly if the subquery is small. Use EXISTS when you only care about existence, not the actual values.

170. Explain the concept of Correlated Subqueries with a practical coding scenario.

Answer:
A correlated subquery is a subquery that references columns from the outer query. It is executed once for each row processed by the outer query.

Example: Find employees who earn more than the average salary of their department:

```sql
SELECT name, salary, department
FROM Employee e1
WHERE salary > (SELECT AVG(salary)
                FROM Employee e2
                WHERE e2.department = e1.department);
```

The inner query uses e1.department, so it is evaluated for each row of the outer query.

172. What is a Clustered Index versus a Non-Clustered Index? How many clustered indexes can a single table have?

Answer:

· Clustered Index: Defines the physical order of data rows in the table. There can be only one clustered index per table (since rows can be sorted in only one order). Typically the primary key is clustered.
· Non-Clustered Index: A separate structure that stores a pointer to the data rows. It does not affect the physical order of the table. A table can have many non‑clustered indexes (up to 999 in SQL Server).

173. What is a Composite Index? Explain the Left-Prefix rule and how column order matters during lookup optimization.

Answer:
A composite index is an index on multiple columns (e.g., (last_name, first_name)).

Left‑Prefix Rule: The index can be used for queries that filter on a prefix of the indexed columns. For (last_name, first_name), queries filtering by last_name or by last_name AND first_name can use the index, but a query filtering only by first_name cannot use the index efficiently. Column order matters: put the most selective columns first.

174. What is an Index Scan versus an Index Seek? Which one indicates an optimized query pipeline?

Answer:

· Index Scan: The database reads the entire index (or a large portion) to find matching rows. It is less efficient, used when many rows are expected.
· Index Seek: The database uses the index tree to navigate directly to the rows that satisfy the query predicate. It is more efficient and indicates an optimized query (especially for selective conditions).

An Index Seek is preferred for point lookups or range queries with high selectivity.

175. How does a database query optimizer use the EXPLAIN or EXPLAIN ANALYZE command to diagnose slow-running queries?

Answer:
EXPLAIN (or EXPLAIN ANALYZE in PostgreSQL, EXPLAIN PLAN in Oracle) shows the execution plan chosen by the optimizer: the order of table access, join methods, indexes used, estimated row counts, and cost.
By analyzing this output, developers can identify:

· Full table scans where an index should be used.
· Suboptimal join orders.
· Missing or inappropriate indexes.
  EXPLAIN ANALYZE also provides actual execution times and row counts, confirming whether the plan matches reality.

176. Explain "Index Covering". How can an index completely eliminate the need to look up data from the base table heap?

Answer:
A covering index contains all the columns needed by a query (both for filtering and returning). When a query’s SELECT list and WHERE clause can be satisfied entirely by the index, the database does not need to access the base table (heap) – this is called a covering index or "index‑only scan". This reduces I/O dramatically.

Example: Index on (department, salary) for a query SELECT department, salary FROM Employee WHERE department = 'IT' – the index covers all columns, so no heap access is needed.

177. What is Table Fragmentation, and how does rebuilding or reorganizing indexes fix database degradation?

Answer:
Over time, as rows are inserted, updated, and deleted, indexes and tables become fragmented – data pages are split, and logical order does not match physical order, causing extra I/O.
Rebuilding an index drops and recreates it, removing fragmentation and reclaiming space.
Reorganizing defragments the leaf level of an index without rebuilding it completely (less intrusive, online).
These maintenance operations improve query performance by making data physically contiguous.

178. How do wildcards (e.g., LIKE '%keyword') render a standard B-Tree index completely useless? What is the alternative?

Answer:
A leading wildcard (LIKE '%keyword') prevents the B‑Tree index from seeking to a starting point because the search is not left‑anchored. The index is scanned fully (or not used at all).
Alternative: Use full‑text search or inverted indexes (e.g., PostgreSQL tsvector, MySQL FULLTEXT) for efficient substring/word searches. Alternatively, store reversed strings and search with LIKE 'keyword%' for suffix matching.

179. What is Database Sharding, and how does it differ fundamentally from Horizontal Partitioning?

Answer:

· Horizontal Partitioning (or partitioning): Splitting a table into smaller physical pieces (partitions) within the same database instance, based on a partition key (range, list, hash). The database still manages all partitions as a single logical table.
· Sharding: Distributes partitions (shards) across multiple database instances (servers). Each shard is a separate database, often located on different machines, to achieve horizontal scalability and high availability.

Fundamentally, sharding is partitioning across servers, while partitioning is within a single server.

180. Explain the operational concept of Database Replication (Master-Slave architecture). How does it optimize read/write distribution?

Answer:
Replication copies data from a master database to one or more slave replicas.

· Master: Handles all write operations (INSERT, UPDATE, DELETE).
· Slaves: Asynchronously replicate changes from the master and serve read queries.

Benefits:

· Read scaling: Distribute read load across slaves.
· High availability: If master fails, a slave can be promoted.
· Backup: Slaves can be used for backups without affecting master performance.

181. What is a Deadlock? How does the database engine detect a deadlock, and how can you structure queries to prevent it?

Answer:
A deadlock occurs when two or more transactions hold locks and each waits for the other to release a lock, causing a cycle.
Detection: The database engine uses a wait‑for graph; if a cycle is detected, it chooses a victim transaction (usually the one with least work) and rolls it back, releasing its locks.
Prevention:

· Access tables/rows in a consistent order across transactions.
· Keep transactions short.
· Use lower isolation levels when possible.
· Use indexing to reduce lock contention (by accessing fewer rows).

182. How do Database Connections Pools (e.g., HikariCP) prevent production latency compared to opening raw database connections per request?

Answer:
Opening a database connection is expensive – it involves network handshakes, authentication, and resource allocation. A connection pool maintains a set of pre‑opened connections that are reused for multiple requests.

· Reduces latency because connection acquisition is fast (just borrowing from the pool).
· Limits the total number of connections, preventing resource exhaustion.
· Avoids the overhead of repeated connection setup.

183. What is a Database View? What is the difference between a standard View and a Materialized View?

Answer:

· View: A virtual table based on a SQL query. It does not store data physically; it executes the query each time it is accessed.
· Materialized View: A view that stores the result set physically (like a table). It can be refreshed periodically or on‑demand. Used for performance with complex queries or aggregations.

Differences:

· Standard view: always up‑to‑date but can be slow.
· Materialized view: faster reads but may be stale; requires refresh.

185. What is a Database Trigger? Give a common use case (e.g., Audit Logging) and explain its performance impact on bulk write operations.

Answer:
A trigger is procedural code that executes automatically in response to certain events (INSERT, UPDATE, DELETE) on a table.

Use case: Audit logging – automatically record changes to a table (who, what, when) in an audit table.

Performance impact: For each row affected by a DML operation, the trigger fires. On bulk operations (large INSERTs), row‑by‑row trigger execution can significantly slow down performance. Consider using statement‑level triggers (if supported) or batching with application‑side logging.

186. Can you execute a transaction commit or rollback statement inside a custom database Trigger? Why or why not?

Answer:
In most databases (Oracle, PostgreSQL, SQL Server), you cannot issue COMMIT or ROLLBACK inside a trigger because the trigger runs within the context of the transaction that fired it. Committing inside a trigger would break the atomicity of the parent transaction and is generally disallowed.
The only exception is when using autonomous transactions (e.g., Oracle's PRAGMA AUTONOMOUS_TRANSACTION), which allow committing independently.

187. What are the key architecture differences between Document stores (MongoDB) and Key-Value stores (Redis)?

· Key‑Value stores (Redis): Simple key‑value pairs with in‑memory storage. Provide fast reads/writes, data expiration, and basic data structures (strings, lists, sets, hashes). Typically used for caching, session stores, real‑time counters.

Differences: Document stores offer query flexibility; key‑value stores prioritize speed and simplicity.

188. Explain the CAP Theorem (Consistency, Availability, Partition Tolerance). Why can a distributed system choose only two?

Answer:
CAP Theorem states that in a distributed data store, you can have at most two of the following three guarantees:

· Consistency: Every read receives the most recent write or an error.
· Availability: Every request receives a (non‑error) response, without guarantee that it contains the most recent write.
· Partition Tolerance: The system continues to operate despite network partitions.

During a network partition, you must choose between consistency and availability. Thus, systems are classified as CP (consistency over availability), AP (availability over consistency), or CA (no partition tolerance, i.e., single‑site).

190. What is Cache Eviction? Explain the difference between Least Recently Used (LRU) and Time-To-Live (TTL) eviction policies.

Answer:
Cache eviction is the process of removing entries when the cache reaches its size limit or entries become stale.

· LRU (Least Recently Used): Removes the entry that was accessed farthest in the past. Good for access‑pattern optimization.
· TTL (Time‑To‑Live): Each entry has a fixed expiration time; it is removed after that time, regardless of access frequency. Good for data that becomes invalid after a certain period.

Redis supports both (via maxmemory and EXPIRE commands).

192. How do you model a One-to-Many relationship in MongoDB (Embedding documents vs Referencing documents)?

Answer:

· Embedding: Store the child documents inside the parent document as an array. Suitable when the child objects are always accessed with the parent and not too large (≤16MB document limit). Example: { user: ..., addresses: [ { ... }, { ... } ] }.
· Referencing: Store child document IDs in the parent, separate collections. Suitable when relationships are many‑to‑many, child documents are large, or need independent access.

Choose embedding for strong consistency and fewer queries; choose referencing for scalability and when data is shared.

193. What is the purpose of an aggregation pipeline in MongoDB? How does it compare to SQL's GROUP BY?

Comparison:

· SQL GROUP BY is limited to grouping and aggregate functions.
· MongoDB aggregation pipeline can perform filtering, reshaping, joining, and even conditional logic in a single pipeline.

194. Explain the concept of eventual consistency in distributed NoSQL databases.

Answer:
Eventual consistency is a consistency model where updates to a distributed system are not immediately visible to all replicas. Given enough time, all replicas will converge to the same value. It is a trade‑off for high availability and low latency.
Example: In Cassandra, a write to one node may take time to propagate to others; reads may see stale data until replication catches up. Used in systems where immediate consistency is not critical (e.g., social media feeds).

Full‑text indexes in relational databases (e.g., MySQL FULLTEXT, PostgreSQL tsvector) build inverted indexes on text columns and support basic relevance ranking, phrase matching, and stopwords. 199. How do database locks work at different levels (Row-level locking vs Page-level locking vs Table-level locking)?

Answer:

· Row‑level locking: Locks only the specific row(s) being modified. Offers highest concurrency but more overhead. Used by InnoDB, PostgreSQL.
· Page‑level locking: Locks a data page (e.g., 8KB) containing multiple rows. Less overhead than row‑level but reduces concurrency. Used by older DBs (SQL Server with page locks).
· Table‑level locking: Locks the entire table. Lowest overhead, but drastically reduces concurrency (writes block all reads/writes). Used for DDL operations or in MyISAM.

Trade‑off: Finer granularity improves concurrency but increases lock management cost. Choose based on workload (OLTP vs OLAP).

200. Write a query to find the top 3 highest-earning employees within each distinct department using a window partition strategy.

Answer:

```sql
WITH RankedEmployees AS (
    SELECT
        employee_id,
        name,
        department,
        salary,
        DENSE_RANK() OVER (PARTITION BY department ORDER BY salary DESC) AS rank_in_dept
    FROM Employee
)
SELECT employee_id, name, department, salary
FROM RankedEmployees
WHERE rank_in_dept <= 3
ORDER BY department, rank_in_dept;
```

This uses DENSE_RANK() to handle ties (if two employees have the same salary, both are counted). Use ROW_NUMBER() if you want strict top N without ties.
