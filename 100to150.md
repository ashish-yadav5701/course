## Topic 3: Full Stack Integration & API Security (Questions 101–150)

### 101. Walk through the complete request-response lifecycle when a React frontend requests data from a Spring Boot backend.

**Answer:**
1. **React UI** triggers an HTTP request (e.g., using `fetch` or Axios) to a backend endpoint.
2. The browser resolves the domain and sends the request to the Spring Boot server.
3. Spring Boot **DispatcherServlet** intercepts the request and routes it to the appropriate controller method based on `@RequestMapping`.
4. The controller method may call a **service layer** to process business logic.
5. The service interacts with the **database** via JPA repositories.
6. The service returns a DTO or entity to the controller.
7. **Jackson ObjectMapper** serializes the Java object to JSON.
8. Spring returns the HTTP response with status code and JSON body.
9. The browser receives the response; React processes it (e.g., in a `useEffect`) and updates the UI.

### 102. What is CORS? Why does it happen when React and Spring Boot run on different ports?

**Answer:**
**CORS (Cross-Origin Resource Sharing)** is a security mechanism that restricts web pages from making requests to a different domain than the one that served the page.  
When React runs on `http://localhost:3000` and Spring Boot on `http://localhost:8080`, the browser blocks the request because the origin (port) differs. The browser sends a preflight `OPTIONS` request to check if the server allows cross-origin requests.

### 103. What are three distinct ways to configure CORS in a Spring Boot application to accept requests from React?

**Answer:**
1. **Using `@CrossOrigin` annotation** on controller or method:
   ```java
   @CrossOrigin(origins = "http://localhost:3000")
   @RestController
   public class MyController { ... }
```

2. Global configuration via WebMvcConfigurer:
   ```java
   @Configuration
   public class CorsConfig implements WebMvcConfigurer {
       @Override
       public void addCorsMappings(CorsRegistry registry) {
           registry.addMapping("/**")
               .allowedOrigins("http://localhost:3000")
               .allowedMethods("*")
               .allowCredentials(true);
       }
   }
   ```
3. Using a CorsFilter bean:
   ```java
   @Bean
   public CorsFilter corsFilter() {
       UrlBasedCorsConfigurationSource source = new UrlBasedCorsConfigurationSource();
       CorsConfiguration config = new CorsConfiguration();
       config.setAllowCredentials(true);
       config.addAllowedOrigin("http://localhost:3000");
       config.addAllowedHeader("*");
       config.addAllowedMethod("*");
       source.registerCorsConfiguration("/**", config);
       return new CorsFilter(source);
   }
   ```

105. Explain the differences between REST API architecture and GraphQL architecture. When would you choose which?

Answer:

· REST: Resource-based, uses HTTP methods (GET, POST, etc.) and status codes. Client gets predefined data structures; often over-fetching or under-fetching occurs.
· GraphQL: Query-based, client specifies exactly what data they need in a single request. Reduces round trips, but adds complexity and caching challenges.

Choose REST when:

· Simple CRUD operations.
· Caching is critical (HTTP caching works well).
· You have a stable, fixed data structure.

Choose GraphQL when:

· Complex data requirements with many nested relationships.
· Mobile apps with limited bandwidth.
· Multiple clients (web/mobile) need different data shapes.

106. How do you implement real-time, bi-directional communication between React and Spring Boot (e.g., WebSockets vs Server-Sent Events)?

Answer:

· WebSockets: Full-duplex, persistent connection. Suitable for chat, gaming, live collaboration. Use Spring’s @EnableWebSocketMessageBroker with STOMP.
· Server-Sent Events (SSE): Uni-directional (server to client) over HTTP. Use SseEmitter in Spring. Simpler for one-way real-time updates (e.g., notifications, stock prices).

Example (WebSocket):

```java
@Configuration

    @Override
    public void registerStompEndpoints(StompEndpointRegistry registry) {
        registry.addEndpoint("/ws").setAllowedOrigins("*");
    }
    @Override
    public void configureMessageBroker(MessageBrokerRegistry registry) {
        registry.enableSimpleBroker("/topic");
    }
}
```

React uses SockJS and stompjs to connect.

107. What is the JSON data format, and how does Jackson ObjectMapper automatically serialize/deserialize objects between Java and JavaScript?

Answer:
JSON (JavaScript Object Notation) is a lightweight text format for data interchange. Jackson is the default JSON library in Spring Boot. It uses reflection to map Java fields to JSON properties. Serialization converts Java objects to JSON; deserialization does the reverse. You can customize with annotations like @JsonProperty, @JsonIgnore, @JsonFormat.

108. Explain the end-to-end user authentication architecture flow using stateless JWT tokens.

Answer:

1. User submits credentials (username/password) to /login.
2. Spring Security authenticates via AuthenticationManager.
3. On success, server generates a JWT with user claims and expiration.
4. Server returns JWT in response body (or cookie).
5. React stores token (usually in memory or local storage).
6. For subsequent requests, React sends token in Authorization: Bearer <token> header.
7. Spring Security filter (JwtAuthenticationFilter) validates token, extracts user details, and sets Authentication in SecurityContext.
8. Requests are authorized based on roles/permissions.
9. Token expires; server responds with 401; client renews token via refresh token flow.

109. Where should you store a JWT token on the frontend (LocalStorage vs SessionStorage vs HttpOnly Cookies)? Compare security vulnerabilities.

Answer:

· LocalStorage: Persistent, accessible via JavaScript. Vulnerable to XSS attacks (if an attacker injects scripts, they can steal the token).
· SessionStorage: Similar but cleared on tab close; also vulnerable to XSS.
· HttpOnly Cookie: Not accessible via JavaScript, so XSS cannot steal it. Protects against XSS but vulnerable to CSRF (unless SameSite=Strict/Lax is used). Recommended for production when combined with CSRF protection.

110. What is an HttpOnly and Secure cookie, and how does it prevent Cross-Site Scripting (XSS) attacks?

Answer:

· HttpOnly: The cookie cannot be accessed via JavaScript (document.cookie), so XSS scripts cannot read it.
· Secure: The cookie is only sent over HTTPS, preventing man-in-the-middle attacks.

By using these flags, even if an XSS vulnerability exists, the attacker cannot extract the JWT from the cookie, mitigating token theft.

112. Explain the concept of Access Tokens vs Refresh Tokens. How does a silent refresh flow work on the frontend?

Answer:

· Access Token: Short-lived (e.g., 15 min), used to authorize API calls.
· Refresh Token: Long-lived, used to obtain new access tokens without re-authenticating.

Silent refresh flow:

1. Frontend detects that access token expired (e.g., 401 response).
2. It sends the refresh token (usually in an HttpOnly cookie or separate storage) to a /refresh endpoint.
3. Server validates the refresh token and issues a new access token.
4. Frontend retries the original request with the new token.
5. If refresh token also expired, user is redirected to login.

113. What happens on the backend when a JWT token expires? How does the Spring Boot backend securely notify the React client?

Answer:
When a token is expired, the JwtAuthenticationFilter will throw an exception. A custom AuthenticationEntryPoint can catch it and return a 401 Unauthorized response with a specific error message like "JWT expired". The frontend intercepts this 401 and triggers the refresh token flow.

Example:

```java
@Component
public class JwtAuthenticationEntryPoint implements AuthenticationEntryPoint {
    @Override
    public void commence(HttpServletRequest request, HttpServletResponse response,
                         AuthenticationException authException) throws IOException {
        response.sendError(HttpServletResponse.SC_UNAUTHORIZED, "JWT expired or invalid");
    }
}
```

115. What is OAuth2, and how does single sign-on (SSO) work when integrating a third-party provider (e.g., Google/Okta) with Java and React?

Answer:
OAuth2 is an authorization framework that allows third-party applications to obtain limited access to user resources without exposing credentials.

SSO flow (e.g., Google):

1. React redirects user to Google's authorization endpoint with client ID and redirect URI.
2. User logs in and consents.
3. Google redirects back with an authorization code.
4. React sends code to Spring Boot backend.
5. Backend exchanges code for access/refresh tokens using client secret.
6. Backend creates a session or JWT for the user and returns it to React.
7. React uses this token for subsequent API calls.

Spring Boot can use Spring Security OAuth2 client to handle the code exchange.

116. What is Cross-Site Scripting (XSS), and how does React natively protect against it? How can a developer accidentally bypass it?

Answer:
XSS allows attackers to inject malicious scripts into web pages. React escapes content inserted into JSX by default, preventing most XSS attacks. For example, {userInput} is rendered as text, not HTML.

Bypass: Using dangerouslySetInnerHTML to render raw HTML from user input, or rendering user-generated URLs in href without sanitization.

117. What is Cross-Site Request Forgery (CSRF)? How do you protect your stateless REST backend from it?

Answer:
CSRF tricks a user into performing unwanted actions on a site where they are authenticated. In stateless JWT setups, CSRF is less of a concern because no session cookie is used (token is sent in header). However, if using cookies, enable CSRF protection with CsrfTokenRepository (e.g., storing token in a cookie with SameSite attribute, or using a double-submit cookie pattern).

118. What is SQL Injection, and how do Spring Data JPA / Prepared Statements inherently prevent it?

Answer:
SQL Injection occurs when user input is concatenated into SQL queries, allowing malicious SQL to be executed. Spring Data JPA uses Prepared Statements (parameterized queries) that treat input as data, not executable code. JPQL and native queries with parameters (?1 or :name) are safe.

119. Explain Content Security Policy (CSP). How does it protect a React single-page application?

Answer:
CSP is an HTTP header (Content-Security-Policy) that restricts resources the browser can load (scripts, styles, images, etc.). It mitigates XSS by blocking inline scripts and only allowing scripts from trusted sources. For React, configure CSP to allow only your domain and necessary CDNs.

120. How do you prevent Denial of Service (DoS) attacks on your Spring Boot API endpoints (e.g., Rate Limiting/Throttling)?

Answer:

· Use Resilience4j RateLimiter or Bucket4j to limit requests per IP/user.
· Implement @RateLimiter annotations or use a filter.
· Alternatively, use a gateway (e.g., Spring Cloud Gateway) with rate-limiting filters.
· Configure server.tomcat.max-threads and connection-timeout to handle overload.

121. Why should you never send sensitive passwords or tokens via HTTP GET query parameters?

Answer:
GET parameters are logged in server logs, browser history, and referrer headers. They are less secure and can be exposed in caches. Use POST with HTTPS for credentials; send tokens in headers (Authorization) or cookies.

122. What is data masking, and how do you ensure sensitive fields (like SSN or credit cards) are masked in Spring Boot logs?

Answer:
Data masking hides sensitive information in logs. Use @JsonIgnore to exclude fields from serialization for API responses; for logs, implement a custom Logback MaskingPatternLayout or use @Masked annotations with a logging filter. Alternatively, use logback-json with structured logging and mask fields at the log appender.

123. What is API Payload Optimization? How do DTOs (Data Transfer Objects) help reduce network payload sizes?

Answer:
API Payload Optimization minimizes the amount of data sent over the network. DTOs contain only the necessary fields, not full entities with lazy-loaded associations, loops, or extra metadata. This reduces bandwidth usage and serialization/deserialization overhead.

125. What is the difference between GZIP compression on the server side and normal uncompressed payloads?

Answer:
GZIP compresses response bodies (text like JSON, HTML) to reduce size, improving network speed. Spring Boot enables GZIP by setting server.compression.enabled=true. Compression reduces bandwidth but adds CPU overhead on server and client (decompression). Most browsers support it.

126. How do you design an API to stream large files (e.g., 500MB PDFs/Videos) from Spring Boot to a React UI without crashing server memory?

Answer:
Use StreamingResponseBody or Resource with InputStreamResource. Example:

```java
@GetMapping("/download")
public StreamingResponseBody streamFile() throws IOException {
    File file = new File("large.pdf");
    return outputStream -> {
        try (InputStream is = new FileInputStream(file)) {
            byte[] buffer = new byte[4096];
            int bytesRead;
            while ((bytesRead = is.read(buffer)) != -1) {
                outputStream.write(buffer, 0, bytesRead);
            }
        }
    };
}
```

This streams the file in chunks without loading the entire file into memory.

```java
public class ApiError {

    private String message;
    private Map<String, String> fieldErrors;

}
```

```java
@ExceptionHandler(MethodArgumentNotValidException.class)
public ResponseEntity<ApiError> handleValidation(MethodArgumentNotValidException ex) {

    ex.getBindingResult().getFieldErrors().forEach(e ->
```

130. Why is it a dangerous security flaw to return raw Java stack traces to the frontend UI inside an API error response?

This information aids attackers in identifying vulnerabilities and planning targeted attacks. Always return generic error messages (e.g., "Internal server error") to clients, and log detailed errors server-side.

132. How do Microservices communicate synchronously (e.g., OpenFeign / WebClient) vs asynchronously (e.g., Kafka / RabbitMQ)?

Answer:

· Synchronous: Client sends request and blocks until response arrives (e.g., HTTP REST, OpenFeign, WebClient). Tight coupling, immediate response, but can cause cascading failures.
· Asynchronous: Client sends message to a broker (Kafka/RabbitMQ) and continues. Decoupled, more resilient, eventual consistency. Best for long-running tasks, event-driven architectures.

133. What is an API Gateway (e.g., Spring Cloud Gateway), and what role does it play in routing frontend traffic to microservices?

Answer:
An API Gateway is a single entry point for client requests. It routes requests to appropriate microservices, handles cross-cutting concerns like authentication, rate limiting, logging, and load balancing. It abstracts the backend complexity from the frontend.

134. Explain the Circuit Breaker pattern (Resilience4j). How does it protect system stability when a downstream service fails?

Answer:
Circuit Breaker monitors failures to a service. When failures exceed a threshold, it "opens" the circuit, failing fast without attempting the call. This prevents resource exhaustion and cascading failures. After a timeout, it allows a few test requests (half-open) to check if the service is healthy. Resilience4j implements this in Spring Boot with annotations.

135. What is Distributed Tracing? How do tools like Spring Cloud Sleuth/Micrometer and Zipkin track a user action from React through 5 microservices?

Answer:
Distributed tracing tracks requests across multiple services. Sleuth adds a unique trace ID and span ID to each request (via headers). Zipkin collects and visualizes these traces, showing the path, timing, and dependencies. React doesn't directly participate; the backend services propagate the trace headers.

136. How do you maintain transactional data consistency across multiple database instances in a microservices setup (Saga Pattern)?

Answer:
The Saga pattern manages distributed transactions via a sequence of local transactions. If a step fails, compensating transactions are executed to rollback. Implement as:

· Choreography: Each service publishes events and listens, coordinating themselves.
· Orchestration: A central orchestrator directs the flow. Use frameworks like Axon, or implement with Spring Boot and Kafka.

137. What is Service Discovery (e.g., Netflix Eureka), and why is it necessary in a dynamic cloud-deployed backend?

Answer:
Service Discovery allows microservices to locate each other without hardcoded IPs. Services register themselves with a registry (Eureka) and clients query it to get dynamic addresses. This is essential in cloud environments where IPs change (auto-scaling, deployments).

142. How does green-blue deployment minimize downtime when pushing upgrades to a production server environment?

Answer:
Green-blue deployment maintains two identical environments: blue (current) and green (new). Traffic is routed to blue. Deploy new version to green, test, then switch traffic to green. If issues, switch back instantly. Zero downtime because the switch is atomic.

149. Explain the concept of Semantic Versioning (SemVer) for REST APIs (e.g., /api/v1/ vs /api/v2/).
